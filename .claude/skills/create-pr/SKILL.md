# How to create a pull request

## Step 1: Gather repository state

Run these commands in parallel:

- `git status --short`
- `git branch --show-current`
- `git log main..HEAD --oneline` (commits that would go into the PR)
- `git diff main...HEAD --stat` (files changed vs main, ignoring main's advances)
- `gh repo view --json owner,name,defaultBranchRef` (confirm the default branch; assume `main` unless the repo says otherwise)

## Step 2: Prepare the branch

- **If the current branch is `main`** (or the repo's default branch): ask the user for a feature branch name (kebab-case, descriptive, e.g. `flux-add-grafana`, `fix-zsh-prompt`, `terminal-macos-ci`). Create it with `git checkout -b <name>`.
- **If there are uncommitted changes**: decide how to commit them.
  - If there are multiple logical units of change, propose splitting them into separate commits and confirm with the user before committing.
  - Stage with explicit paths (never `git add -A` blindly if there are files that may contain secrets).
  - Commit with concise, imperative one-line messages per root `CLAUDE.md` → "Commit & PR Conventions" ("add X", "fix Y", not "added" / "fixed").
  - **Never add attribution, co-author lines, or "generated by Claude" tags** — root `CLAUDE.md` → "Commit & PR Conventions" forbids them.
- **If the branch is not yet on the remote**: push with `git push -u origin HEAD`. Never force push.

## Step 3: Draft the PR title and body

**Title**: ≤70 characters, imperative ("add grafana helmrelease", not "added grafana"). Mentions the main change. If this PR implements a phase of a plan in `docs/work-in-progress/`, include the phase reference (e.g. `(phase 2)`).

**Body** uses this structure:

```markdown
## Summary
- <bullet describing each meaningful change>
- <mention related plan file if applicable, e.g. `docs/work-in-progress/grafana-plan.md`>

## Test plan
- [ ] <how to verify the change, for the area you touched — homelab: `flux get kustomizations`, `flux build` / `flux reconcile`, `kubeconform`, prettier + leading `---` on hand-authored manifests; terminal: `sh terminal/test.sh`, `sh terminal/verify.sh`, `shellcheck --severity=warning terminal/*.sh`; scripts: run the script>
- [ ] <edge cases worth checking>
```

Keep bullets terse and focused on **why**, not every touched line. If a related plan exists in `docs/work-in-progress/`, link to it and note which phase this PR closes.

## Step 4: Show the draft to the user

Before opening the PR, print the proposed title and body and ask for confirmation or edits. This is the last checkpoint before a remote, visible action.

## Step 5: Create the PR

Once the user approves, create the PR **in draft mode by default** with a HEREDOC for the body to preserve formatting:

```
gh pr create --draft --title "<title>" --body "$(cat <<'EOF'
## Summary
- ...

## Test plan
- [ ] ...
EOF
)"
```

Only omit `--draft` if the user explicitly asks for a ready-for-review PR. Return the PR URL to the user.

## Rules

- **Draft by default** — always create PRs with `--draft` unless the user explicitly asks for a ready-for-review PR.
- **No attribution** — never add "Co-Authored-By: Claude" or "Generated with Claude Code" lines to commits or PR bodies. Root `CLAUDE.md` → "Commit & PR Conventions" explicitly forbids it.
- **No force push** — never use `--force` or `--force-with-lease`. Per root `CLAUDE.md` → "Commit & PR Conventions".
- **Confirm before opening** — creating a PR is visible to the team. Always show the draft first and wait for the user to approve.
- **One PR, one purpose** — if the diff spans unrelated changes, propose splitting into separate PRs before proceeding.
- **Reference plans** — if the work implements a phase from `docs/work-in-progress/`, link the plan and note whether this PR closes the phase.
